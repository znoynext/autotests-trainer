# Autotests Trainer

Учебный pet-проект для практики автотестов на C#.

Это локальный тестовый стенд, в котором есть все базовые части реального приложения:

- UI в браузере.
- Авторизация.
- API.
- SQLite-база данных.
- Проект для автотестов на xUnit.
- Подготовка для UI-тестов на Playwright.

Проект можно скачать, запустить на своем компьютере и тренироваться писать автотесты без риска сломать чужую систему.

Демо-доступ:

```text
login: admin
password: admin
```

## Для кого этот проект

Проект сделан для тестировщика, который хочет научиться писать автотесты на C# постепенно и на понятном стенде.

На нем можно тренироваться:

- писать первые автотесты;
- проверять UI через браузер;
- проверять API через код и Postman;
- проверять данные в базе;
- разбираться с авторизацией;
- учиться строить тестовый проект;
- учиться запускать тесты локально;
- учиться вести pet-проект в GitHub.

## Что получится запустить

После установки и запуска у тебя будет локальный сайт:

```text
http://localhost:5171
```

Внутри сайта есть:

- главная страница;
- страница входа;

- создание задач через UI;

- API для работы с задачами;
- локальная SQLite-база, которая создается автоматически.

## Как скачать проект

Открой PowerShell или терминал в папке, где хочешь хранить проект.

Склонируй репозиторий:

```powershell
git clone https://github.com/znoynext/autotests-trainer.git
```

Перейди в папку проекта:

```powershell
cd autotests-trainer
```

Проверь, что ты находишься в корне проекта. Команда должна показать файл `AutotestsTrainer.sln`:

```powershell
dir
```

## Как установить зависимости проекта

Из корня проекта выполни:

```powershell
dotnet restore
```

Эта команда скачает NuGet-пакеты, которые нужны приложению и тестам.

Потом собери проект:

```powershell
dotnet build
```

Если сборка завершилась без ошибок, стенд готов к запуску.

## Как установить браузер для Playwright

После `dotnet build` выполни:

```powershell
powershell -ExecutionPolicy Bypass -File .\tests\AutotestsTrainer.Tests\bin\Debug\net8.0\playwright.ps1 install chromium
```

Эта команда нужна для будущих UI-автотестов. Она скачивает Chromium, которым Playwright будет управлять из C#-тестов.

Если ты пока пишешь только API или интеграционные тесты, сайт будет работать и без этой команды. Но для UI-тестов ее нужно выполнить.

## Что есть в UI

### Главная страница

Обычная стартовая страница стенда. Ее можно использовать для smoke-тестов:

- сайт открывается;
- заголовок отображается;
- ссылки работают;
- незалогиненный пользователь видит нужные действия.

### Страница авторизации

Страница входа в систему.

Можно тренировать тесты:

- успешный вход;
- неверный логин;
- неверный пароль;
- пустые поля;
- проверка сообщений об ошибке;
- проверка редиректа после входа;
- проверка выхода из системы.

### Панель пользователя

Страница, доступная после входа.

Можно тренировать тесты:

- пользователь после входа попадает в закрытую зону;
- незалогиненного пользователя не пускает;
- на странице отображаются нужные элементы;
- ссылки из панели ведут на правильные страницы.

### Список задач

Учебная сущность проекта - задачи.

Можно тренировать CRUD-проверки:

- список задач открывается;
- новая задача создается;
- задача отображается в списке;
- задача закрывается;
- статус задачи меняется;
- обязательные поля валидируются.

### API-консоль

Страница:

```text
http://localhost:5171/api-console
```

Она нужна для ручной проверки API прямо из браузера.

Можно сначала руками дернуть API, посмотреть ответ, а потом написать такой же автотест на C#.

## API проекта

Базовый адрес при локальном запуске:

```text
http://localhost:5171
```

### Получить список задач

```http
GET /api/workitems
```

Что проверять в автотестах:

- статус ответа `200 OK`;
- ответ является JSON;
- в ответе есть массив задач;
- у задачи есть `id`, `title`, `status`;
- после создания новой задачи она появляется в списке.

### Создать задачу

```http
POST /api/workitems
Content-Type: application/json
```

Пример тела запроса:

```json
{
  "title": "Проверить логин",
  "description": "Учебная задача для API-теста"
}
```

Что проверять в автотестах:

- статус ответа успешный;
- созданная задача получила `id`;
- название сохранилось правильно;
- описание сохранилось правильно;
- новая задача появилась в `GET /api/workitems`;
- пустое название не должно проходить, если для него есть валидация.

### Закрыть задачу

```http
POST /api/workitems/{id}/complete
```

Пример:

```http
POST /api/workitems/1/complete
```

Что проверять в автотестах:

- существующая задача закрывается;
- статус задачи меняется;
- повторное закрытие не ломает приложение;
- несуществующий `id` обрабатывается корректно.

## База данных

Проект использует SQLite.

База создается автоматически при запуске приложения. Отдельно устанавливать сервер базы данных не нужно.

Локальный файл базы создается внутри проекта:

```text
src/AutotestsTrainer.Web/App_Data/autotests.db
```

Этот файл не хранится в GitHub. Он локальный и нужен только для твоего запуска.

Что можно тренировать:

- проверить, что после создания задачи запись появилась в базе;
- проверить, что статус задачи изменился после закрытия;
- подготовить тестовые данные перед тестом;
- очистить тестовые данные после теста;
- сравнить данные из API и данные из БД.

Если нужно сбросить локальную базу, останови сайт и удали файлы из папки:

```text
src/AutotestsTrainer.Web/App_Data
```

При следующем запуске приложение создаст базу заново.

## Какие автотесты здесь можно писать

### Smoke-тесты

Самые простые проверки, что приложение живое.

Примеры:

- главная страница открывается;
- страница логина открывается;
- API `/api/workitems` отвечает;
- после запуска доступен базовый маршрут сайта.

### UI-тесты

Пишутся на C# через Playwright.

Примеры:

- открыть сайт;
- нажать ссылку входа;
- заполнить логин и пароль;
- нажать кнопку входа;
- проверить, что открылась панель;
- создать задачу через форму;
- проверить, что задача появилась на странице;
- выйти из системы.

Что полезно освоить:

- локаторы;
- ожидания;
- заполнение форм;
- клики;
- проверки текста;
- Page Object;
- подготовка данных через API перед UI-тестом.

### API-тесты

Пишутся на C# через `HttpClient` или через `WebApplicationFactory`.

Примеры:

- `GET /api/workitems` возвращает `200 OK`;
- `POST /api/workitems` создает задачу;
- ответ API содержит нужные поля;
- API корректно обрабатывает неправильные данные;
- закрытие задачи меняет ее статус;
- несуществующая задача возвращает ожидаемый ответ.

Что полезно освоить:

- HTTP-методы;
- статус-коды;
- JSON-сериализацию;
- DTO;
- негативные проверки;
- подготовку тестовых данных.

### Интеграционные тесты

Проверяют приложение ближе к реальному поведению, но без ручного запуска сайта.

Примеры:

- приложение стартует в тестовом окружении;
- контроллер возвращает успешный ответ;
- API работает вместе с тестовой базой;
- авторизация защищает закрытые страницы;
- после запроса данные сохраняются.

Что полезно освоить:

- `WebApplicationFactory`;
- тестовую конфигурацию;
- тестовую базу;
- изоляцию тестов;
- фикстуры xUnit.

### Тесты авторизации

Примеры:

- вход с правильными данными успешен;
- вход с неправильным паролем неуспешен;
- закрытая страница недоступна без входа;
- после выхода закрытая страница снова недоступна;
- cookie авторизации выдается после успешного входа.

### Проверки базы данных

Примеры:

- после создания задачи появилась запись в таблице;
- после закрытия задачи изменился статус;
- данные из API совпадают с данными в БД;
- тест умеет создавать чистые данные перед проверкой.

### Негативные тесты

Примеры:

- пустой логин;
- пустой пароль;
- неверный пароль;
- создание задачи без названия;
- закрытие задачи с несуществующим `id`;
- запрос к неправильному API-маршруту.

### Регрессионные тесты

Это набор проверок, который запускается после изменений в коде.

Примеры:

- логин работает;
- список задач открывается;
- задача создается через UI;
- задача создается через API;
- задача закрывается;
- база обновляется корректно.

## Рекомендуемый маршрут обучения

### Шаг 1. Ручное знакомство со стендом

Запусти сайт, войди как `admin / admin`, создай несколько задач руками, открой API-консоль.

Цель: понять поведение приложения до написания тестов.

### Шаг 2. Первые smoke-тесты

Напиши проверки:

- главная страница открывается;
- страница логина открывается;
- API задач отвечает `200 OK`.

Цель: научиться запускать тесты и читать результат.

### Шаг 3. Авторизация

Напиши проверки:

- успешный логин;
- неправильный пароль;
- закрытая страница требует вход.

Цель: научиться проверять реальные пользовательские сценарии.

### Шаг 4. API

Напиши проверки:

- получить список задач;
- создать задачу;
- закрыть задачу;
- проверить ошибочный запрос.

Цель: научиться работать с HTTP и JSON.

### Шаг 5. UI через Playwright

Напиши проверки:

- пользователь входит через браузер;
- пользователь создает задачу через форму;
- задача появляется в списке.

Цель: научиться автоматизировать браузер.

### Шаг 6. База данных

Напиши проверки:

- созданная задача есть в SQLite;
- статус закрытой задачи поменялся в базе.

Цель: понять, как связывать UI, API и БД в одной проверке.

### Шаг 7. Улучшение тестового проекта

Когда базовые тесты готовы, можно улучшать структуру:

- вынести повторяющиеся действия в helper-классы;
- сделать Page Object для UI;
- добавить фикстуры xUnit;
- разделить тесты на UI, API, DB;
- добавить генерацию тестовых данных;
- добавить очистку данных после тестов.

## Частые проблемы и решения

### Команда `dotnet` не найдена

Причина: .NET SDK не установлен или терминал был открыт до установки.

Что сделать:

1. Установить .NET 8 SDK.
2. Закрыть PowerShell.
3. Открыть новый PowerShell.
4. Выполнить:

```powershell
dotnet --version
```

### Команда `git` не найдена

Причина: Git не установлен или терминал был открыт до установки.

Что сделать:

1. Установить Git.
2. Закрыть PowerShell.
3. Открыть новый PowerShell.
4. Выполнить:

```powershell
git --version
```

### Сайт не открывается

Проверь, что команда запуска все еще работает:

```powershell
dotnet run --project .\src\AutotestsTrainer.Web
```

Если приложение запустилось на другом адресе, терминал покажет строку вида:

```text
Now listening on: http://localhost:....
```

Открой именно этот адрес.

### Порт занят

Если порт уже занят, останови предыдущий запуск сайта через `Ctrl + C`.

Если не помогло, можно запустить сайт на другом порту:

```powershell
dotnet run --project .\src\AutotestsTrainer.Web --urls http://localhost:5180
```

Потом открыть:

```text
http://localhost:5180
```

### Playwright не запускает браузер

Сначала собери проект:

```powershell
dotnet build
```

Потом установи Chromium:

```powershell
powershell -ExecutionPolicy Bypass -File .\tests\AutotestsTrainer.Tests\bin\Debug\net8.0\playwright.ps1 install chromium
```

### `dotnet test` не может собрать проект, потому что сайт уже запущен

Останови сайт в терминале через:

```text
Ctrl + C
```

Потом снова запусти:

```powershell
dotnet test
```

### Нужно сбросить базу данных

Останови сайт и удали локальные файлы базы из:

```text
src/AutotestsTrainer.Web/App_Data
```

После следующего запуска база создастся заново.

В `.gitignore` исключены:

- `bin/`;
- `obj/`;
- локальная SQLite-база `*.db`;
- временные файлы SQLite `*.db-shm` и `*.db-wal`;
- локальные логи `*.log`;
- пользовательские файлы IDE.

В GitHub должен храниться только код, структура проекта, тестовая инфраструктура и документация. Личные файлы, локальная база и временные артефакты туда не нужны.
